# Tutorial Writing Style

Forge Harness 的教程文档使用中文写作，技术术语在需要精确时保留 English。

教程的语气要像一个工程师在带另一个工程师入门：先用通俗的话建立直觉，再用准确的工程语言说明下一步怎么实现。不要把教程写成项目宣传、架构宣言或参考手册。

## 直接对你说

教程是在和正在阅读、正在动手的人说话。

优先使用：

- `你`
- `我们`
- `这一章`
- `现在先看`
- `下一步会实现`

避免使用：

- `读者会理解`
- `让读者看到`
- `面向读者展示`
- `本文将带领读者`

示例：

```text
差：这一章会让读者理解最直接的 agent loop。
好：这一章先看最直接的一圈 agent loop。
```

## 先讲概念，再讲项目选择

每章先回答当下需要的基础概念，再解释 Forge Harness 为什么这样安排。

推荐顺序：

```text
概念是什么
  -> 最小例子或数据流
  -> 本章要实现什么
  -> 暂时不实现什么
  -> 下一章为什么需要新机制
```

不要一开头就解释项目立场，例如“为什么 Forge Harness 不做完整架构”。先让你知道 `Agent Loop`、`Tool Call`、`Tool Result` 或 `agent harness` 是什么，再谈取舍。

## 先通俗，后工程

一个段落可以先用老师式语气建立直觉，再用工程式语言收束。

示例：

```text
你可以先把 harness 想成 LLM 外面的一圈 runtime。模型负责提出下一步意图，harness 负责把这个意图变成可执行、可记录、可检查的动作。

在 Stage 1，我们只保留一条 execution path：message history -> LLM -> tool call -> bash -> tool result -> message history。
```

通俗解释负责降低门槛。工程语言负责确定边界。

## 每节都要有落点

教程段落不要只解释抽象概念。每节最好给出一个清楚落点：

- 这一节要记住什么。
- 这一节暂时不做什么。
- 这个概念会怎样影响下一步实现。
- 这个痛点会在后续哪类机制里处理。

如果一节只是在赞美某个概念，它通常需要重写。

## 保留必要的 English technical terms

这些词优先保留 English：

- `Agent Loop`
- `Tool Call`
- `Tool Result`
- `Tool Runtime`
- `Context Projection`
- `Observation`
- `Permission Governance`
- `Session`
- `Trace`
- `Runtime State`
- `Verification`
- `Recovery`
- `ChangeSet`

可以用中文解释它们，但不要把 identifier、filename、command、API 名称翻译掉。

## 避免 AI 味

所谓 AI 味，通常不是某个词的问题，而是句子缺少具体对象、具体动作和具体边界。

少用这些写法：

- 大段价值宣言。
- 重复的“不是 X，而是 Y”。
- 没有例子的“逐步构建”“持续演进”“完整体系”。
- “真正关心的是”“足够重要”“核心价值在于”这类空泛句。
- 每段都用同一种句式开头。
- 过度总结读者会获得什么。

优先改成：

- 直接说当前要看哪条数据流。
- 直接说本章会写哪段代码或保留哪条 execution path。
- 用一个短例子解释概念。
- 明确暂时不做的东西。

示例：

```text
差：Forge Harness 的核心价值在于通过渐进式方式让读者理解 agent runtime 的完整能力。
好：这一章只保留一件事：LLM 请求工具，harness 执行工具，再把结果交回给 LLM。
```

## 推荐章节结构

教程章节可以使用这个轻量结构：

```text
# cNN-title

开场：这一章先解决什么。

## 先建立直觉

用通俗语言解释当前概念。

## 看最小数据流

给出 text diagram、命令、代码路径或极小例子。

## 本章实现边界

说明会做什么、暂时不做什么。

## 为什么下一章需要新机制

从当前实现的痛点引出后续章节。

## 进入下一章前

用少量 bullet 总结要带走的判断。
```

这个结构不是硬模板。章节可以调整，但不能丢掉“直觉 -> 例子 -> 工程边界 -> 下一步”的节奏。

## 审稿清单

提交教程文档前，快速检查：

- 有没有把学习者写成第三方 `读者`？
- 章节开头是不是先讲基础概念，而不是项目辩护？
- 每个抽象概念旁边是否有数据流、例子或实现边界？
- 是否有不必要的价值宣言、排比和空泛总结？
- English technical terms 是否保留一致？
- 读完这一章后，是否清楚下一章要做什么？
